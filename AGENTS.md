# Workspace Guardrails

## UI Design Guardrails

- 在实现 UI 之前，先判断这是页面级任务还是组件级任务；不要把页面级的“主体分区 / 大面积视觉主体”规则直接套到输入框、弹窗、表单项这类组件上。
- 做组件级设计时，每个新增的信息元素都必须证明自己在帮助理解当前状态、指导下一步操作或防止错误；标题、说明、标签、统计、预览文案如果不满足这三项之一，默认不加。
- 不要为了“显得完整”叠加多层卡片、边框、阴影或包裹容器；只有存在明确功能分区时，才允许新增视觉层级。
- 在已有主内容容器内，不要再给核心内容套第二层纯装饰性容器；优先让最外层布局承载边界和留白，避免额外边框、圆角或阴影压缩有效内容宽度。
- 默认使用克制的圆角；不要在同一界面里堆叠过多圆角，也不要随意把圆角做大，除非它明确提升层级表达、点击热区或系统一致性。
- 辅助信息要贴近所属对象，并遵循一致的对齐规则；时间、状态、快捷键、说明等元信息不要分散到界面不同角落。
- 交互状态和内容角色必须清晰可分；禁用态、进行中、完成态，以及不同类型内容的区分，不能只依赖微弱的颜色变化。
- 页面标题层级应优先通过排版建立，不要叠加与主标题语义重复的装饰性眉题、小标签或徽标条。
- 导航、侧栏、过滤栏等结构性区域要匹配实际信息密度；内容稀疏时优先压缩宽度，并让折叠或展开入口足够易见。
- 空状态、统计区和摘要区优先先尝试收敛成连续信息带、列表或分隔结构；不要把同层信息默认拆成一组彼此竞争的卡片。
- 桌面视图下，dialog 默认不要超过当前可视页面一半宽度；当内容放不下时，优先通过改成单列/紧凑分组、删除冗余说明和启用内部滚动解决，不要先把弹窗继续放大。
- dialog 标题默认使用区块级标题尺度，不要直接复用页面级超大标题；当弹窗只承载单步输入或确认任务时，标题应明显小于页面主标题。
- 管理页 header 在桌面端应优先收敛为单行操作栏；不要把说明文案、超大标题和主按钮同时堆进首屏，header 高度也不应长期占据页面主要垂直空间。
- 桌面页 header 同时包含“页面身份信息”和“返回/关闭/次级操作”时，优先让标题与关键元信息占据左侧阅读起点，操作按钮退到右侧；不要让操作控件成为第一视觉焦点。
- 当左侧主导航（侧边栏）已经清晰标识当前页面身份时，内容区 header 默认留白并仅保留必要操作；除非存在多级导航路径需要表达，否则不要重复放页面标题。若存在多级路径，使用面包屑承载层级关系。
- 当页面身份信息只包含标题加少量状态/分类元信息时，优先把这些元信息收成贴标题的局部信息块；如果元信息高度很低，可在块内上下排列，但不要因此给整个 header 额外新起一整行。
- 紧凑 header 里局部竖排元信息块的总高度，不应明显超过同排主标题或主按钮这类最高单元；如果竖排会把整行抬高，优先压缩字级、徽标尺寸和间距，必要时改回单行。
- 紧凑 header 里的分类、场景、环境等文字元信息，如果对当前任务没有直接帮助且只是在“填补信息位”，默认删除；不要为了看起来完整而保留低价值文案。
- 对圆角容器做激活态时，不要只加单边描边、单边色条或单边阴影；圆角组件要么使用完整边界系统，要么不强调边界，避免视觉语言自相矛盾。
- 统计摘要区应优先展示“标签 + 关键数值”这类一眼可扫的核心信息；能从上下文推断的解释句默认删除，不要让摘要区比主内容更占高度。
- 卡片或列表项的 header 只承载识别信息和主状态；标签、场景说明、时间等次级信息应下沉到正文区或尾部，避免在第一眼区域堆叠元信息。
- 低优先级元信息要遵循自然阅读顺序，放在内容和主要操作之后；不要把创建时间、更新时间这类信息插入核心信息与主操作之间。
- Dashboard 卡片优先追求“高信息密度但可扫描”：相关字段先尝试压成单行的“标签 + 值”，不要把每个字段都拆成独立段落和大块留白。
- 卡片主 CTA 要落在扫描路径末端，通常位于底部右侧；次级操作降为 outline、ghost 或更多菜单，避免多个并排实心按钮争抢优先级。
- 对多个实例里高度重复、区分价值低的元信息，默认降级到 hover、tooltip、详情页或折叠态，不要长期占据卡片主视觉。
- 卡片底栏如果承载动作，就只承载动作；不要把创建时间、更新时间等元信息继续塞进同一底栏，避免动作区失去视觉重点。
- 卡片、列表项或面板存在右上角工具位时，这个角落默认只留给“更多 / 展开 / 切换 / 固定”等上下文操作；被动状态、时间、标签等展示信息应贴近标题或正文，不要占用工具位。
- 低频、补充性或容错类操作默认收进 overflow/context menu，不与主 CTA 并列占据核心视觉区；主视觉区域优先留给识别信息、关键状态和唯一主动作。
- 未准备好交付的模块默认先隐藏，不要用大面积摘要条、占位列表或“待接入组件”骨架撑页面；只有当占位内容能明确帮助下一步操作时才展示。
- 对明显的单步表单或弹窗，不要默认加入教程式说明、后续流程预告或“创建后会怎样”的复述；当这些文案不影响当前填写决策时，默认删除。
- 字段旁的辅助说明若不是在防止输入错误，就不要常驻显示；能省略就省略，确实需要时再降级为 tooltip、help icon 或按需展开提示。
- 当弹窗标题已经明确指明当前对象或上下文时，不要在正文再重复做一块“当前实例 / 当前对象”摘要；标题已承担识别职责时，正文应直接服务输入。

## Code Structure Guardrails

- `index.ts`、`index.tsx`、`index.js` 这类入口文件默认只保留启动、导出、装配和路由分发职责；具体业务逻辑、窗口创建、IPC 注册、bridge 暴露、类型合同等实现细节应拆到具名模块中。
- 在往现有 `index` 文件继续加代码前，先判断新增内容属于“入口编排”还是“具体实现”；如果它不能被概括为一次导入加一次调用，默认不应继续写在 `index` 里。
- 跨层共享的 channel、payload、result、bridge interface 等合同定义，应优先集中到单独的 contract 模块，避免在 `main`、`preload`、renderer 或其他调用方各自复制一份。
- 在编写一个新的功能点时，请确保有一定的代码拆分，确保拆分符合单一职责原则，确保模块间低耦合；并且考虑一些工具函数是否可以复用，如果可以复用 则需要抽离到utils.ts中

## 对话规范
- 每次代码修改都需要对这次带修改进行简要的文字讲解，包括以下方面：
  1. 为什么这样修改
  2. 修改这段代码怎么起作用
- 当所有代码修改完之后需要进行以下审查：
  1. 这次的代码修改是否需要删除一些重复代码？确保无死代码。
    - 如果涉及到重构，请确保之前的旧代码、无用代码完全清除
  2. 这次代码修改是否可能会引入新的bug？
  3. 这次的代码修改是否存在硬编码？
  4. 这次的代码是否存在过度设计？是否存在设计不足？
